OpenClaw Runtime Walkthrough

把 OpenClaw 还原成一条真实运行链，而不是一堆名词

从第一性原理看，OpenClaw 要解决的是：外部世界不断产生消息、事件、定时任务和设备动作，系统必须把这些输入归到正确的 agent 和正确的会话里，再把需要的上下文、工具和资源装配好，交给模型执行，最后把结果持久化并送回对应位置。

控制中心Gateway统一接收客户端、channel、node 与 cron。
状态边界Session / sessionKey决定历史归属、串行边界和持久化边界。
执行内核Agent Loop把输入变成模型推理、工具动作与最终回复。
可扩能力Tools / Plugins / Channels / Nodes这些不是附件，而是运行时装配的一部分。
阅读开关
Intro

先把“系统到底在干什么”说直白

如果一开始就按 LLM、memory、tool、session、channel 逐项解释，听的人很容易只记住名词，反而看不见主线。OpenClaw 的主线其实很清楚：先接住输入，再决定归属，再执行一次 run，最后把结果和状态都落回系统。

由浅入深 不做类比 以源码为准

第一层：输入从哪里来

输入可以来自 CLI、Web UI、WebChat、macOS app，也可以来自 Discord、Slack、Telegram 这类 channel，还可以来自 node 设备和 cron 定时任务。它们形式不同，但都必须进入 Gateway。

第二层：怎么知道交给谁

OpenClaw 不会直接把消息丢给某个模型，而是先根据来源上下文计算路由，得到 agentId 与 sessionKey。这一步决定后面会使用哪段历史和哪套资源。

第三层：真正跑起来靠什么

当路由和会话确定后，才会进入 agent loop。loop 会装配模型、工具、记忆、工作区和技能快照，然后开始一次完整执行，并把输出与状态同步写回。

一个适合汇报时直接说出口的总句式是： OpenClaw 不是“一个套了提示词的聊天模型”，而是“一个本地控制平面 + 会话状态系统 + agent 执行内核”。模型只是执行环节的一部分，不是整个系统。

Special Flows

特殊流程：为什么 cron、subagent、sessions_send、channel 不是外挂

它们看起来像附加功能，但实际上都是统一运行时的特殊入口或特殊分支。理解这一点，才能把 OpenClaw 看成一个整体系统，而不是功能拼盘。

定时任务：cron

cron 的本质不是“定时跑脚本”，而是“在某个时间点为系统生成新的 agent 输入”。到点后，Gateway 的 CronService 触发 job，再进入标准 agent run 链。

这说明运行时设计很统一：无论输入来自人、来自设备、还是来自时间调度，最后都要转成一次规范化 run。

证据位置
  • src/gateway/server.impl.ts:94-116 Gateway 启动时组装 cron service。
  • src/cron/service.ts:7-60 CronService 暴露 start、list、add、run、wake 等能力。

子代理：subagent

subagent 的本质不是线程，也不是普通函数调用，而是“从当前 run 派生一个独立会话”。子代理有自己的 sessionKey、自己的 token 消耗、自己的完成状态，结束后再把结果通知回父会话。

这样设计的好处是边界清楚：主会话与子会话不会互相污染，子任务可以并行，也能独立记录和清理。

证据位置
  • docs/tools/subagents.md:12-13 子代理运行在自己的 session 中。
  • src/agents/tools/sessions-spawn-tool.ts:82-85 明确是 spawn isolated session。
  • src/agents/subagent-registry.ts:155-220 维护状态并回写 session 信息。

会话间通信：sessions_send

一个 agent 想把任务交给另一个 session，并不是直接修改对方历史，而是通过 Gateway 再发起一个新的 agent run。也就是说，会话间通信仍然走统一控制平面和权限校验。

这能避免任意 session 越权访问别人的上下文，也能保证所有 run 都接受相同的可见性和策略约束。

证据位置
  • src/agents/tools/sessions-send-tool.ts:35-63 最终通过 callGateway({ method: "agent" }) 发起 run。
  • src/agents/tools/sessions-send-tool.ts:83-131 做跨 agent 策略校验。

Channel 监听器

Discord、Slack、Telegram 监听器并不直接调用模型。它们负责接住外部事件、提取路由上下文，再把输入归并成 OpenClaw 内部的会话和 run。

也就是说，channel 不只是“接入插件”，它是现实世界进入运行时的入口层。

证据位置
  • extensions/discord/src/monitor/listeners.ts:13-15 监听器依赖 resolveAgentRoute。
  • docs/concepts/architecture.md:12-18 所有 messaging surface 都统一由 Gateway 拥有。
判断某个能力是不是运行时本体，有一个很简单的标准： 只要它最终会创建、定位、串行化或回送一次 agent run，它就属于运行时本体，而不是外围附属功能。cron、subagent、sessions_send、channel 都符合这个标准。
Persistence

数据和文件落点：系统把状态写到哪里

想真正讲明白运行时，必须把“进程里一时存在的东西”和“写盘后能恢复的事实”区分开。OpenClaw 的很多关键设计，都是围绕“哪些东西必须落盘”来做的。

会话索引

Gateway 是 session 的唯一真实来源。每个 agent 都有自己的 session store，典型路径是 ~/.openclaw/agents/<agentId>/sessions/sessions.json。这里存的是 sessionKey 到元数据的映射，不是全文历史。

证据位置
  • docs/concepts/session.md:57-69

会话全文转录

每个 session 的消息历史以 transcript 形式持久化，典型文件是 ~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl。模型要继续对话，靠的是这份历史，而不是靠进程内存。

记忆文件

OpenClaw 明确把 memory 定义为工作区里的 Markdown。短期日记常写在 memory/YYYY-MM-DD.md，长期稳定事实写在 MEMORY.md。这说明记忆是可见、可编辑、可审计的系统事实。

证据位置
  • docs/concepts/memory.md:11-28

记忆索引

记忆源头是 Markdown，但为了让 agent 做语义检索，系统会建立 memory index manager，管理 embedding provider、向量与关键词检索、文件监控和增量同步。

证据位置
  • src/memory/manager.ts:80-144

技能快照

新 session 启动时，技能会构建 snapshot 并写回 session entry。这样后续运行就知道这段会话当时看到的技能集合是什么，避免上下文漂移。

证据位置
  • src/agents/agent-command.ts:908-938

子代理状态

子代理状态也不是只留在内存里。registry 会把运行记录持久化，并把 startedAt、endedAt、runtimeMs、status 等信息回写到对应 session entry。

证据位置

对老板来说，这一节特别重要：OpenClaw 许多看起来“智能”的行为，本质上都是靠明确状态持久化来保证可恢复、可解释、可继续，不是靠进程里偷偷留着一些变量。
Design Choices

运行时设计取舍：为什么 OpenClaw 要做成这样

真正理解一个系统，不只是知道它怎么做，还要知道它为什么不换一种更简单的做法。下面这些取舍，几乎决定了整个运行时的形状。

为什么要有 Gateway

如果没有 Gateway，每个 UI、每个 channel、每个 node 都要自己维护连接、状态、权限和 session，这会导致状态分裂。OpenClaw 把 Gateway 设成唯一控制平面，就是为了让“谁在说话、说到哪了、该用哪个 agent、结果发回哪里”只有一个地方说了算。

为什么 run 要按 session 串行

同一个会话如果同时跑两个 loop，就会出现工具冲突、历史顺序错乱、token 统计不准、最终状态不一致。OpenClaw 明确把每个 session 当成一个串行执行通道，这是为了保证一致性。

证据位置
  • docs/concepts/agent-loop.md:31-35,45-49

为什么 memory 要落到 Markdown

因为要让记忆成为用户和 agent 都能审计、编辑、迁移的事实，而不是模型内部不可见的隐含状态。这样 memory 才能被搜、被读、被修改，也才能跨运行恢复。

为什么 tools 不能全部放开

工具不是越多越好。OpenClaw 有 catalog、profile、session visibility、agent-to-agent policy，这说明它把工具看成运行时权限面，而不是普通函数库。工具一旦能跨 session、跨 agent、连设备，就必须受控。

证据位置
  • src/agents/tool-catalog.ts:27-220
  • src/agents/tools/sessions-send-tool.ts:83-131
更深入一点：为什么 OpenClaw 的模块看起来很多
因为它把几个通常会混在一起的问题硬拆开了。路由只负责“落到谁”；session 只负责“历史和边界”；loop 只负责“这次 run 如何执行”；tool 管理只负责“这次 run 能用什么能力”；memory 只负责“哪些信息可跨轮保留”；channel 只负责“外部输入如何进来”。模块多，理解成本上升，但边界更稳定。
更深入一点：为什么 channel、UI、cron 都能接到同一内核
因为 OpenClaw 在输入层做了统一抽象。无论输入来自手工消息、网页、Discord 事件还是定时器，本质上都要被还原成一次规范化 run 请求，带上 agentId、sessionKey、消息内容和上下文。入口统一，后面的运行链就能复用。
Talk Track

给老板讲时可以怎么说

下面这部分不是再讲新知识，而是帮你把前面的内容压缩成适合口头汇报的表达方式。可以直接照这个顺序讲。

一句话版

OpenClaw 的运行时可以理解成一个本地控制平面。它把多种输入统一接进来，用路由和 session 把上下文边界分清，再把模型、工具、记忆、工作区等资源装配成一次 agent run，最后把结果流式返回并写入持久化存储。

三句话版

  1. 第一句：所有入口最后都进 Gateway，所以状态不会散在各处。
  2. 第二句：sessionKey 决定这次输入属于哪个上下文，也是串行执行和持久化的边界。
  3. 第三句：agent loop 会把模型、工具、记忆、资源装好，执行一次完整 run，并把结果再送回原来的地方。

老板最该理解的点

  • LLM 不是整个系统，LLM 只是 loop 中的推理引擎。
  • Memory 不是“模型自己记住了”，而是写入工作区文件并可检索。
  • Session 不是聊天窗口概念，而是运行时状态边界。
  • Tool、channel、cron、subagent 都不是外挂，而是统一运行时的接口层。

如果老板问“为什么这么复杂”

因为 OpenClaw 不是只做单轮问答，而是要长期运行、接多个入口、支持多 agent、支持设备和自动化，还要保证状态一致、能恢复、能审计、能扩展。复杂度主要来自状态管理，而不是来自提示词。

整体结论：OpenClaw 的核心不是某个“很聪明的 prompt”，而是一套把外部输入、会话边界、模型执行、工具能力、持久化状态和回送链路组织在一起的运行时。向老板解释它，最有效的方式不是一项项介绍功能，而是沿着“输入进入 Gateway，落到 session，进入 loop，调用工具，写回状态，再把结果送出”这条链讲。
