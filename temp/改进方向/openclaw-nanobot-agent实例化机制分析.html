Instance Design · Workspace · Identity · Session

OpenClaw 与 nanobot 的 agent 实例化机制，到底差在哪

这份分析只盯住一个主题：一个 agent 在系统里是怎么“被定义、被选中、被赋予工作空间、被装配人格、被延续状态”的。结论先说：nanobot 更像“配置驱动的单实例壳”，openclaw 更像“具名 agent 节点系统”。

一句话结论

nanobot 先把 “config + workspace” 跑顺，重点是轻量实例；openclaw 先把 agent 变成正式配置实体，再给它独立的 workspace、agentDir、identity、sessions，重点是长期稳定编排。

nanobot：轻、快、单实例友好 openclaw：重、稳、多 agent 友好 对 agent-alpha：先学 nanobot，再补 openclaw

这次看的不是

不是模型聪不聪明，也不是功能多少，而是 agent 实例的工程边界：谁决定它是谁、它在哪工作、它的状态放哪、子 agent 算不算新的实例。

workspace 是 nanobot 的主边界 agentId + workspace + agentDir 是 openclaw 的主边界
nanobot 方案 openclaw 方案 核心差异 该借什么
Route A

nanobot：配置驱动的单实例壳

nanobot 的实例，不是一个显式的 `AgentSpec`。它更像“选中一个配置文件，再把 workspace 灌进 AgentLoop”。实例边界主要来自 config 和 workspace。

  • `agents.defaults.workspace` 是实例最关键的空间边界。
  • CLI 用 `--config` 选择实例，用 `--workspace` 做一次性覆盖。
  • workspace 会自动补齐 `AGENTS.md`、`SOUL.md`、`USER.md`、`TOOLS.md`、`memory/`、`skills/`。
  • `ContextBuilder` 会从 workspace 根目录直接读取这些文档，拼成 system prompt。
  • 子 agent 共享同一个 workspace，本质更像后台 worker，不是正式子实例。

Route B

openclaw：具名 agent 节点系统

openclaw 的实例不是“顺手传一个 workspace”，而是配置里的正式 agent 实体。它先有 agentId，再有 `workspace`、`agentDir`、`sessions`、`identity`，最后才是 runtime 调度。

  • agent 在配置里有独立条目：`id / name / workspace / agentDir / model / identity / tools / subagents / runtime`。
  • 系统会通过 `agentId / sessionKey / bindings / default agent` 解析当前实例。
  • `workspace` 和 `agentDir` 是分开的，多个 agent 不能共享 `agentDir`。
  • workspace bootstrap 更完整，包含 `IDENTITY.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 等。
  • session 默认按 `agentId` 隔离，落到 `agents/<agentId>/sessions/`。
  • 子 agent 不是后台线程，而是真正新的 session 实例，带 `spawnedBy / spawnedWorkspaceDir / spawnDepth / status` 等元数据。

Comparison

实例化核心差异

维度 nanobot openclaw
实例入口 `config + --workspace` `agentId + sessionKey + bindings`
实例定义方式 默认实例为主，运行时覆盖 agent 是配置一级对象
人格装配 workspace 根目录文档整体注入 workspace 文档 + identity 结构化层
状态边界 主要围绕 workspace workspace、agentDir、sessions 三层拆开
子 agent 共享 workspace 的后台执行体 新的子 session / 子实例
适合阶段 单实例标准化 多 agent 长期演进

把两者翻译成一句人话

nanobot 像是“给每个实例一张办公桌，再把规则贴在墙上”；openclaw 像是“先建组织架构，再给每个 agent 独立办公室、档案柜、工牌和会话流水”。

哪边更像你现在需要的

如果你现在就想把 `agent-alpha` 从“能跑”推进到“边界清楚”，更该先借 nanobot 的实例壳；如果你下一步要真正做多 agent 编排、角色隔离、子 agent 生命周期，那就要继续往 openclaw 的方向补。

Recommendation

对 agent-alpha 最值得借的顺序

Step 1

先学 nanobot

把实例最少收成：`config/profile + workspace + bootstrap docs + session scope`。先把“输入工作路径、自动加载人格文档”彻底做稳。

Step 2

再学 openclaw

补上 `agentId`、`agentDir`、`IDENTITY.md`、per-agent sessions，让实例成为系统中的正式对象，而不是一段临时运行状态。

Step 3

最后做编排

让子 agent 变成真正的新实例：有 `spawnedWorkspace`、`parent_session`、`spawnDepth`、`status`，而不是共享父工作区的后台线程。

最稳的路线不是“选一个抄到底”，而是先借 nanobot 把单实例做标准，再借 openclaw 把实例做成编排系统里的正式节点。
