Harness Engineering Review

OpenClaw 与 learn-claude-code 的多 Agent 机制分析

这份报告不讨论“模型本身聪不聪明”，只讨论多 agent 背后的外壳工程。 重点是把两个项目在入口、编排、运行时基础设施、工作区、持久化和长期运行方式上的差异讲清楚， 并进一步回答一个更实际的问题：这些差异，为什么会直接影响到 agent-alpha 未来该怎么演进。

一句话结论

learn-claude-code 更像一套“教学型分层演进模型”： 它把 Claude Code 风格的多 agent 机制一层一层拆开，让你看见这些机制为什么存在。 openclaw 更像一套“产品级长期运行 runtime”： 它关心的不只是 spawn 子 agent，而是 route、session、workspace、gateway、cron、heartbeat、delivery 这些系统级部件如何长期协同。

learn-claude-code：抽象清楚 openclaw：控制面完整 task = control plane workspace = agent instance root

这份报告的阅读方式

如果你更关心“怎么借鉴”，优先看“关键判断”“逐维度对比”和“对 agent-alpha 的启发”。 如果你更关心“证据从哪里来”，最后一节提供了本次静态阅读中重点查看的源码入口。

Method

1. 这份分析在回答什么

这里的“多 agent 机制”不是泛指“能不能开子线程”或者“能不能调用另一个模型”， 而是更接近 harness engineering 里的系统问题：一个 agent 什么时候被创建、以什么身份运行、挂在哪个工作区上、 如何被路由、如何被回收、它的上下文和状态放在哪里、以及这些能力是通过教学式叠加出来，还是通过平台式控制面来管理。

分析范围
  • 入口：用户如何进入多 agent 世界，入口层有没有承担控制面角色。
  • 编排：子 agent 是临时委派，还是受管的 session run。
  • 运行时：有没有 gateway、daemon、cron、heartbeat、恢复机制。
  • 工作区：workspace 是纯 cwd，还是 agent 实例的根。
  • 持久化：任务、消息、会话、运行状态落在哪里。
  • 对 agent-alpha 的借鉴：哪些可以学，哪些不该直接照搬。
判断基线

这次不是做“谁更先进”的排行榜，而是看它们各自解决的真实问题是什么。 用最直白的话说：learn-claude-code 教你理解机制， openclaw 教你运营机制。

Overview

2. 总体判断：两者不是同赛道上的“轻重版”，而是两种不同目标

如果只看表面，很容易把这两个项目理解成“一个简单版，一个复杂版”。但更准确的说法是： 它们要解决的问题不同，所以多 agent 机制的形状也不同。前者强调概念演进，后者强调系统长期运转。

维度 learn-claude-code openclaw 本质差异
项目目标 教学型拆解，逐步搭出 Claude Code 风格机制 产品型 runtime，长期运行与多入口接入 一个更像课程，一个更像平台
多 agent 核心 在简单 loop 外叠加任务、团队、自治层 把 agent 视为有身份、有会话、有路由的受管对象 一个是“机制叠层”，一个是“系统编排”
任务语义 任务板是主结构之一 session run 与 route 承担更核心角色 一个偏 task-centric，一个偏 session-centric
工作区语义 最终发展到 worktree 隔离，但更像执行侧设施 workspace 从一开始就是 agent 身份根目录的一部分 一个把工作区作为执行隔离，一个把工作区作为 agent 容器
长期运行能力 通过 inbox、task board、idle loop 模拟自治 通过 gateway、heartbeat、cron、registry、恢复机制支撑 一个是教学性自治，一个是运营级自治
对 agent-alpha 的价值 教你如何收束抽象边界 教你如何补齐控制面和生命周期 前者适合学“原则”，后者适合学“骨架”
Entry Layer

3. 入口层：它们是怎么把用户带进多 agent 世界的

入口层的差异很关键，因为它决定了多 agent 是“代码里的一种技巧”，还是“系统里的一种正式能力”。 入口越像控制面，后续的路由、恢复、治理和运行隔离就越容易成立。

learn-claude-code

它的入口不是一个厚重的控制面，而是一套逐步升级的教学脚本。你会看到 s04_subagent.py、s07_task_system.py、 s09_agent_teams.py、s11_autonomous_agents.py、 s12_worktree_task_isolation.py 这样的阶段式文件。

  • 入口是按能力阶段演进的，不是一个统一、长期运行的控制平面。
  • 每一层都尽量不破坏 core loop，而是在 loop 外补一层 harness。
  • 这很适合学习，因为你能看见机制为什么一步步长出来。
openclaw

它的入口更像操作系统里的“程序入口 + 管理入口”。在 src/cli/program/build-program.ts 里先搭出 CLI 上下文， 再由 register.agent.ts、register.onboard.ts 等命令注册模块把入口能力装配起来。

  • agent 命令更像一次运行入口。
  • agents 命令更像实例管理入口。
  • gateway-cli/run.ts 与 run-loop.ts 则把入口延展成长期运行服务。
这一步最值得记住的一点是： learn-claude-code 先让你理解“为什么需要多 agent 机制”， openclaw 则直接回答“多 agent 机制上线后该由谁来管”。
Orchestration

4. Agent 编排：子 agent 是一次函数式委派，还是一个受管运行对象

这部分是两者差异最大的地方。表面上都能“开子 agent”，但背后的语义并不一样。 一个更像把复杂问题切给另一个执行者，一个更像启动一条受控的运行实例。

learn-claude-code 的编排路线
  • s04： 子 agent 最早是轻量级并发工作者。核心语义是“开一个新上下文，干完回来交摘要”。
  • s07： 任务系统出现后，agent 编排开始围绕 task board 发生，任务具备依赖关系与状态。
  • s09： agent team 出现，编排不再是临时调用，而是有角色、有成员、有消息收件箱的团队协作。
  • s11： 自主 agent 引入 idle loop，agent 能够轮询 inbox、认领任务，朝自治系统靠近。
  • s12： 最重要的升级是把 task 与 worktree 分离，开始把“控制面”和“执行面”拆开。
openclaw 的编排路线
  • sessions-spawn-tool.ts 说明 spawn 不是私有技巧，而是正式工具接口。
  • spawn 时不仅传 prompt，还会带上 runtime、mode、thread、sandbox、resumeSessionId 等运行参数。
  • subagent-spawn.ts 与 subagent-registry.ts 说明子 agent 是受 registry 管理的 run 记录。
  • action-spawn.ts 则把对话中的“我要开个子 agent”接到正式编排动作上。
  • 这意味着子 agent 天然挂在 session lifecycle 里，而不是只作为临时 helper 存在。
最核心的差异

在 learn-claude-code 里，子 agent 先是“能分工的工作者”，再逐步长成团队成员。 在 openclaw 里，子 agent 从一开始就是“受管的运行实例”。 这件事会直接影响后面的恢复能力、权限控制、生命周期管理和跨入口一致性。

Runtime

5. 运行时基础设施：哪个更像教学 loop，哪个更像长期运行系统

如果你把多 agent 看成“系统”，那真正拉开差距的不是 prompt，而是运行时基础设施。 有没有 daemon、有没有 registry、有没有锁、有没有恢复、有没有后台周期任务，这些决定了工程上能不能长期稳定运转。

learn-claude-code：运行时故意保持朴素

它的强项在于把复杂度控制住。即使到了 team、autonomous、worktree 阶段， 核心 loop 依然保持清晰，额外机制通过 inbox、task board、json 文件和 polling 慢慢叠加。 这种设计特别适合学习和验证概念，因为你不会被太厚的 runtime 外壳淹没。

openclaw：运行时是一级公民

gateway-cli/run-loop.ts 展现出明显的服务化特征： 任务 drain、嵌入运行 drain、锁、优雅关闭、进程内重启与完整重启都被明确考虑。 infra/heartbeat-runner.ts、 cron/isolated-agent/run.ts 进一步说明它不是“用户来一次我跑一次”， 而是默认系统会自己活着、自己定期做事。

结论：两者解决的是不同阶段的问题

如果你还在探索机制边界，learn-claude-code 的方式更利于保持心智简单。 如果你已经需要多个入口、多个身份、长期运行、故障恢复和后台自治， openclaw 的那套 runtime 骨架会更接近真实生产需求。

Workspace

6. 工作区模型：工作目录到底是“执行路径”，还是“agent 实例根目录”

这一层和你的目标非常相关，因为你明确希望每个 agent 类都可以输入工作路径、自动识别人格文档，并用于多 agent 编排。 这里恰好是 openclaw 最值得借鉴的地方。

learn-claude-code 的工作区理解

在前期阶段，工作区更多是执行时共享或隔离的文件系统环境。到了 s12_worktree_task_isolation.py，工作区终于被明确抬到结构层， 但它仍然更偏执行面：重点是避免不同任务互相污染，而不是定义 agent 身份容器。

  • 它强调 worktree 隔离的必要性。
  • 它把“task”和“worktree”分开，是非常重要的架构启发。
  • 但它没有像 openclaw 那样，把工作区与 agent identity、prompt docs、工具边界天然绑在一起。
openclaw 的工作区理解

src/agents/workspace.ts 说明 workspace 从一开始就不只是 cwd。 它更像某个 agent 实例的根目录：身份、工具、文档、记忆、引导材料和运行痕迹都会围绕这个根展开。

  • 能识别 AGENTS.md、SOUL.md、TOOLS.md、IDENTITY.md、MEMORY.md 等文件。
  • 这让“给 agent 一个目录，就让它拥有一套人格和行为边界”成为自然能力。
  • 这也是为什么它更容易支持多实例、多身份和长期会话管理。
对 agent-alpha 来说，这里的真正启发不是“支持更多 md 文件”， 而是把 workspace 升格成 agent instance root。这样一来，人格发现、skill 装载、MCP、日志、session、私有配置才有共同归属点。
State

7. 持久化与通信：任务板、收件箱、会话存储，各自扮演什么角色

多 agent 机制只要超过一次性 demo，就一定会碰到两个问题：状态放在哪里，协作怎么传递。 两个项目都回答了这个问题，但回答方式不同，因此系统行为也不同。

learn-claude-code：任务视角

s07_task_system.py 里任务以文件形式落在 .tasks/*.json，依赖关系和状态变化都围绕任务板来组织。 到了团队和自治阶段，JSONL inbox 变成 agent 间通信的主载体。

openclaw：会话视角

在 openclaw 里，session 是更核心的单位。消息由 route 决定去向，session store 负责保存语义连续性， subagent registry 负责 run 生命周期，announce/delivery 决定结果如何回送。

差异落点

前者更适合清晰表达“谁在做哪件事”，后者更适合承载“这个 agent 现在处于什么运行状态、和谁关联、还能不能恢复”。

问题 learn-claude-code 的回答 openclaw 的回答
谁负责协调工作 task board 与 team protocol route + session + runtime registry
消息怎么传 per-agent inbox / JSONL channel routing 与 delivery
状态怎么恢复 依赖任务文件、收件箱和 loop 逻辑重建 依赖会话存储、run registry、gateway 生命周期管理
自治从哪里来 idle loop 轮询 + auto-claim task cron、heartbeat、隔离运行与 session lifecycle
Implication

8. 对 agent-alpha 的启发：不该照搬哪一套，应该抽取哪一层

你的目标不是复刻一个更复杂的 openclaw，也不是把 learn-claude-code 当教材原样拼进项目里。 更现实的做法是：用前者收束控制面，用后者校准抽象边界。

最该学 learn-claude-code 的地方
  • 保持 core loop 简洁，不要把所有能力塞进一个巨型 runtime。
  • 把任务、团队、自治、工作区隔离这些概念明确拆开，不要在代码里糊成一团。
  • 把 task = control plane 这个思想吸收进来，让调度和执行分层。
  • 优先让机制边界干净，再去叠加能力密度。
最该学 openclaw 的地方
  • 把 workspace 升格成实例根目录，而不是只当 cwd。
  • 把 spawn 子 agent 做成正式 runtime 对象，而不是临时 helper 调用。
  • 把 session、registry、恢复、路由、心跳这些控制面能力留出位置。
  • 入口层从“跑一段逻辑”升级为“管理一次运行”。
最值得收束成三条原则

第一，workspace = agent instance root。 第二，task = control plane，执行器只是执行面。 第三，subagent run = managed lifecycle object，这样多 agent 才能真正走向可恢复、可治理、可持续运行。

如果现在就照搬 openclaw，会发生什么

很可能会把 agent-alpha 重新带回复杂化。因为 openclaw 的强项恰恰在于控制面完整， 但完整控制面也意味着更多抽象层、更多状态类型、更多入口和更多运维语义。 对还在收束架构的项目来说，直接整包复制通常会让系统先变重，再变乱。

如果只学 learn-claude-code，又会缺什么

你会比较容易得到一个讲得清楚、结构干净的系统，但当你开始需要 session 恢复、MCP 生命周期、 agent 身份目录、定时自治或多入口接入时，控制面会显得偏薄，很多能力还需要你自己再补一轮。

更稳妥的路线

先用 learn-claude-code 的思想把概念边界切清楚，再选择性引入 openclaw 的几块骨架：workspace root、spawn registry、session-based lifecycle、轻量 routing。 这样更符合你“减少冗余、优化架构、提高配置灵活性和鲁棒性”的目标。
