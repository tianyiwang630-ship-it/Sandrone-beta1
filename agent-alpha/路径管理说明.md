# agent-alpha 路径管理说明

这份文档说明 `agent-alpha` 当前的路径重定向、目录布局、环境变量注入，以及 Python / npm / Go 的运行边界。

## 1. 总体思路

当前 `agent-alpha` 的路径管理可以概括为：

- Python 运行环境尽量收敛到 `agent-alpha/.venv`
- 常见用户目录、配置目录、缓存目录、临时目录重定向到 `agent-alpha` 项目内部
- skill 采用双路径：主路径 `agent-alpha/skills`，兼容安装路径 `agent-alpha/home/.agents/skills`
- npm / pnpm / yarn / Go 保留宿主机全局语义，不强制重定向到 `agent-alpha`
- sandbox 的重点是禁止写出 `agent-alpha` 和当前 workspace 之外

## 2. 启动时发生什么

环境准备脚本是 [setup-agent-alpha.ps1](/D:/files/demo/0312-newagent/agent-alpha/setup-agent-alpha.ps1)。
日常启动脚本是 [start-agent-alpha.ps1](/D:/files/demo/0312-newagent/agent-alpha/start-agent-alpha.ps1)。

首次起环境或依赖变更时，运行 `setup-agent-alpha.ps1`，主要做这些事：

1. 把当前目录切到 `agent-alpha`
2. 检查 `uv` 是否可用
3. 如果不存在 `agent-alpha/.venv`，创建 Python 3.12 虚拟环境
4. 要求 `.venv` 的 Python 版本是 `3.12`
5. 调用 `agent.core.runtime_paths` 创建运行目录并生成运行环境变量
6. 将生成的环境变量注入当前进程
7. 安装依赖和 editable 包：

```text
uv pip install --python .venv/Scripts/python.exe -r requirements.txt
uv pip install --python .venv/Scripts/python.exe -e .
```

日常启动时，运行 `start-agent-alpha.ps1`，主要做这些事：

1. 把当前目录切到 `agent-alpha`
2. 确保存在 `agent-alpha/.venv`
3. 要求 `.venv` 的 Python 版本是 `3.12`
4. 调用 `agent.core.runtime_paths` 创建运行目录并生成运行环境变量
5. 将生成的环境变量注入当前进程
6. 执行：

```text
python -m agent.cli.main
```

这意味着 `agent-alpha` 自身始终假设跑在 `.venv` 中，但日常启动不会重复安装依赖。

## 3. 环境变量重定向

XDG_CACHE_HOME        -> agent-alpha/cache
XDG_DATA_HOME         -> agent-alpha/data
XDG_STATE_HOME        -> agent-alpha/state

Windows 下还会额外设置：

```text
HOMEDRIVE

```

## 4. 持久环境变量

持久环境变量文件是：

```text
agent-alpha/config/runtime_env.local.json
```

这个文件用于保存：

- token
- cookie
- API key
- auth header
- proxy
- 其他服务配置

它会在每次 bash 执行前注入运行环境。

但它不能覆盖路径策略变量。当前被保留、不可从这个文件覆写的变量包括：

- `HOME`
- `USERPROFILE`
- `PATH`
- `TEMP`
- `APPDATA`
- `AGENT_ALPHA_ROOT`
- `XDG_*`
- `VIRTUAL_ENV`
- 以及缓存目录相关变量

## 5. PATH 规则

当前 PATH 前缀顺序是：

1. `agent-alpha/.venv/Scripts` 或 `.venv/bin`
2. `agent-alpha/bin`
3. 已存在的 `agent-alpha/home/.local/Python*/Scripts`
4. 原系统 PATH

因此，`agent-alpha` 会优先找到：

- `.venv` 里的 Python
- `.venv` 里装出来的 CLI，例如 `twitter.exe`、`agent-reach.exe`
- `agent-alpha/bin` 下的本地命令入口
- `pip install --user` 等方式安装到 alpha-local Python user base 的 CLI

- `state/browser/*`：browser 工具未来使用的状态目录

## 7. skill 路径

当前 skill loader 读取两层路径：

```text
主路径：agent-alpha/skills/<skill-name>
兼容路径：agent-alpha/home/.agents/skills/<skill-name>
```

优先级规则是：

- 如果两个地方都有同名 skill
- `agent-alpha/skills/<skill-name>` 优先

因此可以理解为：

- `agent-alpha/skills` 更适合正式内置 skill 或覆盖版本
- `agent-alpha/home/.agents/skills` 更适合第三方安装的 skill

## 8. Python 环境

当前 Python 路径策略是：

- alpha 自己运行在 `agent-alpha/.venv`
- Python 包安装优先进入 `.venv`
- Python CLI 也优先进入 `.venv`
- `pip` 缓存进入 `agent-alpha/cache/pip`
- `uv` 缓存进入 `agent-alpha/cache/uv`

当前文档和系统提示词的推荐规则是：

```text
agent-alpha/.venv/Scripts/python.exe -m pip install ...
uv pip install --python agent-alpha/.venv/Scripts/python.exe ...
```

也就是说，规范上已经把 `.venv` 作为 Python 和 Python CLI 的首选目标。

## 9. npm / pnpm / yarn / Go

这几类工具目前保留宿主机全局语义，不做项目内强制收口。

当前策略是：

- `npm install -g ...` 允许
- `pnpm add -g ...` 允许
- `yarn global add ...` 允许
- `go install ...` 允许

当前没有专门重定向这些变量：

- `NPM_CONFIG_PREFIX`
- `GOPATH`
- `GOBIN`
- `GOCACHE`
- `GOMODCACHE`

所以它们的全局安装结果仍然可能落到宿主机自己的全局目录，而不是 `agent-alpha` 内。

## 10. bash 工具运行规则

bash 工具定义在 [bash_tool.py](/D:/files/demo/0312-newagent/agent-alpha/agent/tools/bash_tool.py)。

当前行为是：

- 命令执行目录固定为 `agent-alpha`
- 每次执行前重新构造 runtime env
- 运行 shell 时带上项目内重定向后的环境变量

所以 bash 不是直接拿系统终端的原始环境运行，而是始终带着 `agent-alpha` 自己的环境运行。

## 11. sandbox 与路径边界

路径边界主要由 [path_policy.py](/D:/files/demo/0312-newagent/agent-alpha/agent/core/path_policy.py) 和 [sandbox_guard.py](/D:/files/demo/0312-newagent/agent-alpha/agent/core/sandbox_guard.py) 控制。

当前规则可以概括为：

- `workspace` 内：读写允许
- `agent-alpha` 项目内：读允许，绝大多数写允许
- `agent-alpha` 里的受保护路径：写入需要确认
- `agent-alpha` 外部：读取允许，写入和删除拒绝

当前明确受保护的项目路径只有：

```text
agent/core
agent/tools
```

外部已安装 CLI / exe 的规则是：

- 允许执行 `agent-alpha` 外部的已安装 CLI / exe
- 允许调用 npm / Go 全局工具位置里的命令
- 禁止通过参数、重定向、复制、移动、删除等方式显式写入 `agent-alpha` 和当前 workspace 外部

例如：

```text
C:/Users/<user>/go/bin/agent-browser.exe --version
C:/Users/<user>/AppData/Roaming/npm/agent-browser.cmd open https://example.com
```

这类“只执行外部工具”的命令允许。

```text
C:/Users/<user>/go/bin/agent-browser.exe --output C:/Users/<user>/Desktop/page.json
agent-browser > C:/Users/<user>/Desktop/page.json
```

这类“明确写外部文件”的命令会被拒绝。

也就是说，现阶段 sandbox 最核心的目标是：

- 不让 agent 直接写到 `agent-alpha` 外部
- 不让 agent 直接改核心运行代码，除非明确批准

## 12. 当前状态下需要注意的点

1. Python 的规范已经偏向 `.venv`，但历史上仍可能留下 `pipx` 或 `tools/uv` 的旧痕迹。
2. npm / Go 还没有内收进 `agent-alpha`，仍按宿主机规则工作。
3. `agent-alpha/home`、`agent-alpha/cache`、`agent-alpha/state` 这些目录本质上属于运行态目录，不等于源码目录。
4. browser 工具已经预留了 `state/browser/*` 目录，但具体怎么用还需要后续工具实现来决定。

## 13. 一句话总结

当前 `agent-alpha` 的路径管理是：

- Python 和大多数运行时目录尽量项目内化
- skill 有双路径但主路径优先
- npm / Go 继续走宿主机全局语义
- sandbox 主要防止写出 `agent-alpha` 外部
