Harness Engineering Review

openclaw、nanobot 与 agent-alpha 架构对比

这份报告不是在比较“谁功能最多”，而是在比较三者如何搭建 agent 的“执行底盘”。 也就是：任务怎样进入系统，模型和工具怎样被组织起来，会话怎样持续，出错后怎样恢复，以及未来怎样支撑多 agent 编排。

生成日期：2026-04-08 比较视角：当前现状 + 下一步演进 输出重点：对 agent-alpha 的收敛建议

一句话结论：openclaw 像“成熟操作系统级底盘”， nanobot 像“研究友好的轻量内核”， agent-alpha 则更像“把技能、MCP、权限和上下文机制拼出来的第一代工作台”。

对你的意义

  • 如果目标是“尽快稳定跑起来、好理解、好改”，agent-alpha 更应该靠近 nanobot，而不是直接追 openclaw 的体量。
  • 如果目标是“以后多 agent、跨工作区、人格自动识别、能力插件化”，又不能只做 nanobot 式极简，需要保留 agent-alpha 现在已经有的模块边界意识。
  • 所以最合适的方向不是二选一，而是“以 nanobot 的简洁主链路为骨架，吸收 openclaw 的少量关键制度化能力”。
Platform-first

openclaw

核心思想是先搭一个稳定控制面，再把 agent、渠道、插件、UI、节点设备都接到这张网里。

  • 优点：能力边界清晰，复杂场景能控，适合长期演进。
  • 代价：子系统非常多，理解和裁剪成本高。
  • 适合借鉴：控制面、工具策略、工作区治理、多 agent 会话模型。
Loop-first

nanobot

核心思想是先把“单个 agent 如何稳定跑完一轮任务”做短、做清楚，再逐步外接能力。

  • 优点：主链路短，调试容易，研究和教学价值高。
  • 代价：复杂治理能力相对薄，越往大系统走越容易补丁式增长。
  • 适合借鉴：最小可用 loop、工具注册、provider 抽象、内存降级策略。
Integration-first

agent-alpha

核心思想是围绕 Skills + MCP + 本地工具构造通用执行框架，并加入工作区、权限、上下文压缩等能力。

  • 优点：方向和你的目标一致，模块意识比“纯脚本式 agent”更强。
  • 代价：主链路还偏重，很多能力是“各自成立”，但还没有收束成一条特别干净的 runtime spine。
  • 适合演进：减少重复层、统一运行时边界、把人格与工作区装配做成一等概念。

判断基线：从 harness engineering 的角度看，真正决定系统上限的不是“有多少工具”，而是“这些工具和上下文、权限、会话、执行循环之间的关系是否被统一抽象”。这正是三者差异最大的地方。

Core Matrix

核心对比矩阵

维度 openclaw nanobot agent-alpha
架构重心 控制面优先，agent 只是整个运行网络里的一个节点。 单 agent loop 优先，先把“消息进来后怎样跑完”做短。 能力集成优先，围绕 Skills、MCP、本地工具拼出统一框架。
入口形态 CLI、Gateway、UI、节点设备、多渠道入口并存。 CLI + 渠道总线为主，结构更集中。 单 agent CLI 为主，当前更接近本地工作台。
模型抽象 模型目录、鉴权轮换、回退策略都较完整。 provider registry 简洁直接，足够扩展但不过度治理。 LLM profile 已经分离出来，但和 runtime 主链的耦合仍偏紧。
工具抽象 核心工具 + 插件工具 + policy 共同决定最终工具面。 ToolRegistry 很直接，默认工具注册逻辑好理解。 ToolLoader 功能很强，已含 MCP、内建工具、skill loader、tool search。
会话与上下文 会话、路由、转录、压缩、渠道绑定都有制度化设计。 SessionManager + memory consolidation 构成轻量双层记忆。 ContextManager 有压缩能力，但历史、日志、会话索引目前仍比较分散。
权限与安全 策略层最成熟，owner-only、tool allowlist、运行时守卫都成体系。 偏实用型保护，够用但不强调复杂策略治理。 已有 sandbox_guard + permission_manager，但交互式权限流程还不够统一。
MCP 集成 把 MCP 当平台能力的一部分，校验和边界更严格。 懒连接 MCP，包装成原生工具，成本较低。 MCP 是一等特性，还做了 searchable/core 分类，方向很好。
多 agent 能力 天然强，session、lane、agent scope 都是专门设计过的。 有 subagent，但更像辅助执行功能。 已支持多工作区和 role 配置，但距离完整多 agent 编排仍差一个统一编排层。
适合作为下一步参照 适合学“制度”和“边界”，不适合整套照搬。 适合学“主链瘦身”和“模块最小闭环”。 应该成为“nanobot 骨架 + openclaw 局部治理能力”的混合体。

复杂度评级

这里说的不是功能多少，而是“理解一个改动要穿过多少层”。

openclaw：高。因为很多能力已经平台化，任何改动都可能碰到会话、插件、路由、安全联动。

nanobot：低到中。主链很短，定位问题的路径通常也短。

agent-alpha：中。比 nanobot 重，但还没重到 openclaw 那种平台化密度。

演进潜力评级

这里关注的是“未来是否容易支撑你要的人格、多工作区、多 agent”。

openclaw：很高，但代价是搬运成本也很高。

nanobot：中等偏上，前期快，后期需要补治理层。

agent-alpha：高。因为它已经在“通用框架化”方向上了，只是需要瘦身和抽象收束。

读法建议：如果只看一页，这一页已经够你把三者的定位差异抓住；后两页是把“为什么会这样”拆开讲。

Deep Dive

从 Harness Engineering 角度逐维拆解

01 / 运行入口与控制面

谁在“指挥”系统

openclaw 的主语不是单个 agent，而是 Gateway 控制面。很多动作先落到控制面，再分发给 agent、渠道、UI、节点。

  • openclaw：更像“操作系统 + 网络总线”。
  • nanobot：更像“一个能自己完成任务的小型执行器”。
  • agent-alpha：现在还主要是“单机工作台”，控制面意识刚开始出现。

启示：如果 agent-alpha 未来要多 agent 编排，需要先把“运行时装配层”和“任务执行层”分开。

02 / 模型与工具接线方式

谁负责把能力接进来

nanobot 的做法是最朴素也最利于维护的：provider registry 管模型，ToolRegistry 管工具，AgentLoop 只消费这两个结果。

  • openclaw：工具不只是“注册”，还受插件、策略、会话环境共同塑形。
  • nanobot：接线简单，心智负担小。
  • agent-alpha：ToolLoader 很强，但有“一个类扛了太多编排责任”的趋势。

启示：agent-alpha 下一步最值得做的是把 ToolLoader 拆成“发现、选择、运行、策略”几层。

03 / 会话、上下文、记忆

谁在维护任务连续性

openclaw 强在“会话是平台对象”；nanobot 强在“记忆是最小闭环”；agent-alpha 现在是“上下文压缩有了，但会话治理还没完全成型”。

  • openclaw：会话不仅是历史，还和路由、发送策略、模型覆盖绑定。
  • nanobot：SessionManager + MemoryConsolidator 够轻，也有失败兜底。
  • agent-alpha：压缩逻辑有价值，但 session history、log、workspace temp 的职责边界还可再清晰。

启示：比起继续加压缩功能，agent-alpha 更该先统一“什么算会话真相源”。

04 / 安全、权限、隔离

谁决定哪些动作能做

openclaw 把安全当成架构前提；nanobot 把安全当成实用保护；agent-alpha 已经意识到权限问题，但现在还是“有组件，未完全成体系”。

  • openclaw：tool policy、owner-only、边界读取、运行时守卫很系统。
  • nanobot：限制够用，优先保证主链不失控。
  • agent-alpha：sandbox_guard、permission_manager 都有，但权限决策入口比较分散。

启示：要支撑多 agent，就不能让每个工具自己决定安全规则，必须收口到统一 policy 层。

05 / 可扩展性与平台化程度

未来能否持续长大

openclaw 已经是平台；nanobot 还是产品级骨架；agent-alpha 正在平台化早期。

  • openclaw：适合“很多能力长期共存”。
  • nanobot：适合“先把 80% 核心能力做干净”。
  • agent-alpha：已经有 Skills、MCP、Role、Workspace 这些平台胚子。

启示：agent-alpha 不缺方向，缺的是“收束秩序”。

06 / 多 Agent 编排准备度

距离你的目标还有多远

你想要的是“每个 agent 可以输入工作路径、自动识别人设 md、便于多 agent 编排”。这个目标上，三者都能提供不同层面的启发。

  • openclaw 提供的是：agent scope、workspace、session lane 的组织方法。
  • nanobot 提供的是：subagent 能力先做成简单工具，再慢慢升级治理。
  • agent-alpha 已有多 workspace 和 role 配置，但“agent definition”还不是第一类对象。

关键缺口：agent-alpha 还缺一个专门的 AgentSpec 或 AgentDescriptor 层，用来统一描述人格、工作区、工具组、模型和权限。

这一页的核心判断：openclaw 赢在“体系化”，nanobot 赢在“主链清爽”，agent-alpha 当前最大的问题不是功能不够，而是“运行时脊柱还不够单纯”。一旦主链变干净，你想要的工作区人格化、多 agent 编排才会真正自然。

Actionable Direction

agent-alpha 的现状判断与建议收敛路径

Current Diagnosis

agent-alpha 现在更像什么

它已经不是“玩具 agent”，因为它有：

  • 系统提示词构造层
  • Skills 装配层
  • MCP 自动发现与分类加载
  • 上下文压缩
  • 工作区与角色过滤
  • 权限与 sandbox 保护意识

但它也还不是“成熟底盘”，因为这些能力现在更像“并列存在”，而不是被一条极清楚的 runtime pipeline 串起来。

  • ToolLoader 承担了太多职责。
  • ContextManager 很重要，但它和 session/log/memory 的总关系还不够统一。
  • RoleConfig 已出现，但抽象还偏薄，离“agent 配方”还有距离。
Target Shape

更适合往哪种形态收敛

最合适的目标不是“缩成 nanobot”，也不是“长成 openclaw”。更准确地说，是：

做一个 nanobot 风格的主链 + openclaw 风格的边界治理层。

  • 主链保持短：输入请求 → 解析 agent 配方 → 组装上下文 → 选工具 → 执行 loop → 落会话。
  • 治理层外置：权限、MCP 分类、工具组策略、工作区边界、日志归档，不直接塞进 loop。
  • 人格与工作区一等化：让 agent 不再只是 runtime 参数，而是一个明确的配置实体。

优先借鉴 nanobot 的部分

  • 把 AgentLoop 缩成真正的“任务执行循环”，不要让它感知太多外围治理细节。
  • 把 provider、tools、memory、session 拆成小而完整的闭环模块。
  • 在关键失败路径上优先做降级，而不是先做复杂策略。

优先借鉴 openclaw 的部分

  • 工作区与人格启动文件的边界规则。
  • 工具策略层，而不是把权限判断散落在工具里。
  • agent scope / session scope 的概念分离。
Roadmap A

第一阶段：先做“主链瘦身”

  • 抽出 AgentSpec：统一描述 `workspace`、`persona docs`、`role`、`model profile`、`tool policy`。
  • 把 ToolLoader 拆成 4 层：`ToolDiscovery`、`ToolCatalog`、`ToolPolicyResolver`、`ToolExecutorRouter`。
  • 把 ContextManager 从“压缩器”升级为“上下文装配器”，让 summary、history、workspace docs、skills 都有固定入口。
  • 把 session 真相源固定下来，避免日志、history、session temp 各说各话。
Roadmap B

第二阶段：再做“多 agent 装配”

  • 每个 agent 输入一个工作路径后，先在根目录自动识别 `AGENTS.md`、人格文档、可选 `SOUL.md`。
  • 让 AgentSpec 支持“继承默认配置 + 局部覆盖”，便于编排多个 agent。
  • 引入轻量 AgentRegistry，不一定要做成 openclaw 那样重的控制面，但至少要能统一创建、列出、启动和调度 agent。
  • 多 agent 之间传递的不是自由文本，而是结构化任务对象 + 最小必要上下文。

最终建议：agent-alpha 未来最应该成为“面向个人/小团队编排的模块化 agent runtime”。它不需要一开始就做成 openclaw 那样的大平台，但必须尽快从“功能集合”升级为“有统一装配逻辑的 harness”。这一步一旦完成，你想要的冗余减少、架构优化、配置灵活性、鲁棒性和多 agent 人格化，都会顺很多。

报告依据：已对比三者的入口层、运行时、工具注册、上下文/会话、MCP 接线、权限/策略相关核心文件与 README 描述，结论偏重 harness engineering 视角，而非功能全量盘点。
