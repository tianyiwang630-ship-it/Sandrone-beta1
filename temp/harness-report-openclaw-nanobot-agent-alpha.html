Report Summary

从 Harness Engineering 角度，对比 OpenClaw、nanobot 与 agent-alpha

这份报告不比较“模型谁更聪明”，而是比较三个项目作为 agent 外壳工程时，谁更像平台、谁更像骨架、谁更适合你当前这条“从旧项目简化并走向多 agent 编排”的路线。

日期：2026-04-09 视角：Harness Engineering 形式：汇报型正式评审材料 对象：OpenClaw / nanobot / agent-alpha
一句话结论
最值得借鉴的是 nanobot，最不该整体照搬的是 OpenClaw。

OpenClaw 的强项是“平台级控制面与生命周期工程”； nanobot 的强项是“轻量但闭环的 agent harness”； agent-alpha 已经有不错的 coding-agent 内核，但目前更像“可运行 runtime”，还没完全长成“可长期演进的工程外壳”。

OpenClaw：平台能力最强 nanobot：简化路线最贴合 agent-alpha：最接近真实落地点
对你现在最重要的问题，不是“谁做得最强”，而是“哪一套 harness 思路能让 agent-alpha 在不重新变重的前提下，逐步走向更稳、更清晰、更多 agent 编排”。
本报告回答什么
  • 三个项目在 harness engineering 上分别处于什么层级。
  • 它们在入口、配置、会话、工具加载、生命周期、测试策略上的差异。
  • 为什么 nanobot 最适合作为你当前阶段的直接参考样板。
  • 为什么 OpenClaw 应该“选择性借鉴”，而不是整体模仿。
  • agent-alpha 下一步最该补哪几层外壳。
Page 02
Evaluation Method

比较方法与评价维度

这里的“harness engineering”可以理解为：一个 agent 项目作为工程外壳，是否能被稳定地启动、配置、扩展、隔离、恢复、测试与演进。换句话说，比的是“车架和底盘”，不是“发动机参数表”。

维度 1

运行编排

看入口是否清楚，主循环是否稳定，工具与子 agent 的调度是不是围绕明确的运行时来组织。

维度 2

工程支撑

看配置、路径、日志、权限、会话、恢复、健康检查、部署入口这些外围工程是否成体系。

维度 3

可演进性

看未来做多 agent、按工作目录运行、自动识别人设文档、扩展更多能力时，是否容易落地而不失控。

评价项 重点看什么 如果做得弱，会发生什么
入口与生命周期 是否有清楚的启动、停止、重启、恢复和运行模式区分。 项目能跑，但一到中断、重启、长时间运行就开始变脆。
配置与路径模型 配置是否从代码常量走向实例化配置，路径是否围绕工作区或 agent 实例组织。 一做多实例、多 agent 或环境切换，就容易互相污染。
上下文与身份装配 AGENTS.md、SOUL.md、skills、memory 等是否围绕工作目录自动装配。 人格、规则、技能会停留在“项目级共享”，不利于编排。
状态与会话 会话如何存、如何恢复、如何隔离，是否支持长时间运行与历史维护。 上下文污染、恢复失败、历史不可追溯。
测试护栏 测试是否覆盖“工程外壳最容易坏”的地方，而不是只测几个工具函数。 表面能跑，迭代时却很容易回退或打碎已有流程。

注：这份报告基于关键骨架文件与目录结构的静态比对，不包含运行压测结论。

Page 03
Project Positioning

三个项目各自像什么

这三者并不处于同一层级。如果不先区分“平台”“骨架”“当前运行时”这三个角色，很容易把结论做偏。

Platform-Scale Harness

OpenClaw

更像一个完整平台，而不是一个“单 agent 项目”。它已经有控制面、gateway、daemon、插件、channel、CLI 命令树、onboarding 和运维外壳。

定位：平台级外壳

  • 优势在“控制面完整”，不是在“代码少”。
  • 适合学习边界设计、生命周期治理、插件契约、命令系统。
  • 不适合作为当前阶段的整体模仿对象，否则很容易把项目重新做重。
Lightweight Closed Loop

nanobot

像是从 OpenClaw 中提炼出来的一套轻量骨架。它保留了会话、配置、路径、频道、cron、subagent 等外壳要件，但体量小得多。

定位：轻量闭环骨架

  • 最适合研究“如何做小，但不散”。
  • 对工作区、人设文档、上下文装配比较友好。
  • 很适合作为 agent-alpha 的直接借鉴对象。
Coding Runtime in Transition

agent-alpha

更像一个已经具备 coding-agent 实战能力的运行时核心：有 MCP、skill、tool loader、权限、session snapshot，但还没把外围外壳彻底做扎实。

定位：可运行 runtime

  • 核心能力并不弱，甚至很接近你的真实需求。
  • 问题主要不在“不会做事”，而在“怎么更稳、更清晰、更可编排”。
  • 后续最该补的是外壳层，而不是继续堆能力点。
如果把 OpenClaw 比作一座机场，nanobot 更像一座结构紧凑但流程完整的小型航站楼，而 agent-alpha 现在更像一架已经能飞、但地面保障体系还没补齐的飞机。

Page 05
Deep Dive

逐项分析：谁强在什么地方

下面的分析会更细一些，适合在“结论大体明白以后”继续往下看，帮助判断 agent-alpha 具体要学什么、不学什么。

一、入口与主控层

OpenClaw：最像完整产品外壳

OpenClaw 的命令树是系统化设计的，不是把几个脚本揉在一起。它有 setup、onboard、gateway、agent、agents、status、doctor、backup 这些分层命令，说明它已经把“用户如何进入系统”和“系统如何被运维”当成一等公民。

对 harness engineering 来说，这意味着它不只是能跑，而是知道怎么被安装、怎么被诊断、怎么被远程调用、怎么承接不同运行模式。

对比结论

nanobot 与 agent-alpha 还停留在轻量入口

nanobot 已经有统一 CLI，所以不散；但它不是平台级命令面。agent-alpha 目前则更偏向“直接跑 agent”，CLI 里混合了不少会话和工作区管理职责。这种结构在项目早期很正常，但一旦要接更多运行模式，会先感到吃力的是 CLI，而不是模型。

二、配置与路径组织

nanobot：最适合借的点

nanobot 的一个重要优点，是很多运行时目录都从“当前配置上下文”推导出来，例如 data、logs、cron、workspace。这让它天然更容易支持多实例与独立工作区，不需要把很多路径硬写在项目根上。

对你来说，这正好对应你的目标：每个 agent 类可以输入工作路径，并自动识别对应的人格文档与上下文资源。

对比结论

agent-alpha 现在最大的结构性短板之一

agent-alpha 虽然已经引入 workspaces，但仍有明显的“项目根目录中心化”痕迹。比如 skills、mcp-servers、一些系统配置仍然默认从项目根取。这样做在单实例阶段没问题，可一旦进入多 agent 编排，就会面临共享资源边界不清的问题。

三、工作区与人格装配

nanobot 的启发比 OpenClaw 更直接

nanobot 会把工作区中的 AGENTS.md、SOUL.md、USER.md、TOOLS.md 作为系统提示的重要组成部分。这件事非常贴近你的目标，因为你明确提出：希望每一个 agent 类都能输入工作路径，并自动识别 agent 人格 md 文档，便于多 agent 编排。

对比结论

agent-alpha 已经起步，但还没彻底“实例化”

目前 agent-alpha 已经会读取首工作区的 AGENTS.md 与 SOUL.md，这说明方向是对的。但它还只是“把文件读进来”，还没有进一步把 identity、skills、memory、tool scope、private config 组合成一个明确的 agent instance contract。

四、会话、记忆与状态

OpenClaw 强在治理，nanobot 强在轻量闭环

OpenClaw 的 session 与 routing 更偏平台治理能力；nanobot 则用比较小的实现，把会话保存、历史裁剪、memory/consolidation 这些关键环节做成闭环。这种“够用且好改”的状态模型，对你的当前阶段反而更有价值。

对比结论

agent-alpha 目前更偏 snapshot，而不是完整状态体系

agent-alpha 已经有 session snapshot 和 CLI 恢复，这是好事。但整体上更像“为了恢复当前对话而保存”，而不是围绕长时间运行、隔离、演进建立完整状态系统。也就是说，它更像恢复功能，而不是状态架构。

Risk And Opportunity

重点对象：agent-alpha 现在最像什么，最缺什么

这一页只谈 agent-alpha。因为对你来说，真正有价值的不是“别人多强”，而是“当前手里这套东西，最该怎么往前推”。

已有优势

它的内核并不弱

  • 已经有面向 coding agent 的核心骨架：tool loader、MCP 自动发现、skills、权限、session snapshot。
  • 已经有多工作区的雏形，不是完全写死单目录。
  • 已经开始支持从工作区读取 AGENTS.md、SOUL.md，这非常接近你的人格自动识别目标。
  • 测试并不是空白，说明它具备继续重构的安全基础。

结论：可演进基础是有的

主要短板

外壳层仍然不够“实例化”

  • 配置仍偏代码常量化，而不是面向实例与环境的配置模型。
  • 项目根目录仍承担太多共享资源角色，未来容易形成耦合。
  • CLI 中混入较多工作区和恢复逻辑，后续接入 API 或 orchestrator 时会先卡在边界不清。
  • 总线、cron、生命周期治理还偏薄，更像占位能力而不是真正的运行时治理层。

结论：问题主要在工程外壳，不在 agent 核心

风险 1

过早平台化

如果现在直接向 OpenClaw 靠齐，先做大而全的 gateway、控制面、复杂插件系统，项目很可能会先进入“框架建设期”，而不是“需求驱动的简化期”。

高风险

风险 2

继续项目根目录中心化

如果 workspaces 只是“允许传一个目录”，但 skills、persona、memory、mcp 仍主要围绕项目根，那么多 agent 编排只会停留在表层。

高风险

机会

先把实例边界做对

一旦 agent-alpha 先把 agent instance contract 做清楚，后面不管接 FastAPI、调度器、多 agent orchestration 还是 UI，都有稳定基座可以接。

高价值机会

来源 最值得借的东西 为什么值得借 不建议照搬的部分
nanobot 工作区中心化、人格文件装配、轻量 session/memory 闭环、简洁配置路径模型 最贴合“把旧项目简化并保持可扩展”的目标 主循环内部过于集中时，后面会影响更复杂编排
OpenClaw 生命周期治理、命令层分层、restart/drain 思想、插件契约意识 能帮助你避免以后一做大就散架 大 gateway、重平台控制面、过于完整的产品壳
agent-alpha 自身 MCP 与 coding-agent 方向积累、现有工具系统、已有测试与权限思路 这是你最贴近真实需求的资产，不应该推倒重来 继续让项目根目录承担过多运行时职责
最适合 agent-alpha 的路线，不是“朝 OpenClaw 扩张”，而是“以 nanobot 的轻量骨架为参考，把现有 coding-agent 核心重新挂到更清楚的实例外壳上”。

Recommended Path

对 agent-alpha 的建议路线：先补外壳，再补编排

下面这条路线不是“终极架构”，而是更适合你当前阶段的推进顺序。重点是先解决最容易影响后续扩展的结构问题。

阶段 1

先做 agent instance contract

定义“一个 agent 实例到底由什么组成”：workspace、identity docs、skills、memory、mcp sources、private config、session store、logs。

  • 目标是让“输入工作路径，自动识别人格文档”成为正式能力，而不是散落逻辑。
  • 这一层做对后，多 agent 编排才有真正的承载对象。
阶段 2

拆 CLI 与运行时边界

把 CLI 的交互逻辑、session 恢复、workspace 管理，从核心 runtime 中进一步剥离，让 runtime 能同时服务 CLI、FastAPI 接口与未来的 orchestrator。

  • 这一步会直接改善后端接口设计空间。
  • 也会降低以后引入多种入口时的耦合成本。
阶段 3

补生命周期与编排层

再向 OpenClaw 选学：健康检查、运行状态、重启恢复、任务排队、中断处理、多 agent orchestration。

  • 这时借 OpenClaw 才不会显得“太重”。
  • 因为底下已经先有清楚的 agent 实例边界了。
优先级最高

实例边界

先回答“这个 agent 是谁、住在哪、读什么、写什么、跟谁隔离”。

优先级中等

运行入口分层

让 CLI、API、cron、subagent 编排都能挂在同一个核心 runtime 上。

优先级靠后

平台能力补全

健康检查、运维工具、重启治理这些该做，但不该先做成平台壳。

最终判断

你现在最该学的是“nanobot 的轻骨架”，而不是“OpenClaw 的重控制面”

因为你当前任务不是从零搭一个产品平台，而是从既有项目中减冗余、清边界、提模块化、提鲁棒性，并把工作区和人格识别能力做扎实。

收束成一句话

先从“能跑”走到“能稳、能隔离、能编排”

OpenClaw 适合学“做大以后怎么不失控”，nanobot 适合学“做小以后怎么不散”，而 agent-alpha 现在最需要的，是把自己从“能跑”推进到“能稳、能隔离、能编排”。
