For AI Vibe Coders

GitHub 链接不是魔法入口，真正起作用的是 skill 结构、工具体系、安装链路、session 快照和运行时注入。

从源码和文档还原后，更准确的说法是：OpenClaw 内建了一套可扩展运行时，它能扫描 skill 资产，判断资格，按需要安装依赖，把结果做成 session 快照，再把这些内容以模型能消化的方式注入到一次次 agent run 里。

结论一 GitHub 链接只是入口 真正被系统接住的是仓库中的 SKILL.md、references、安装说明、plugin 资产和可执行工具路径。
结论二 skill 不是功能本体 skill 更像操作知识层，负责把“什么时候做、怎么做、依赖什么工具”告诉 agent。
结论三 运行时会做过滤和压缩 OpenClaw 不会把所有 skill 原文一股脑塞给模型，而是先筛选、排序、截断，再做成 prompt 和 snapshot。
结论四 这是一整套机制，不是单点功能 skill、tool、plugin、channel、workspace、session、gateway 和 loop 是一起工作的。
阅读控制
Intro

先把一句最重要的话说清楚

OpenClaw 并没有一个“任意 GitHub 仓库都能自动变成 skill”的通用魔法按钮。更准确的事实是：它有一套能识别、 接纳、安装、过滤、注入扩展资产的运行时设计。只要某个仓库的内容刚好符合这套机制，比如仓库里已有 skill 目录、SKILL.md、references、frontmatter 元数据、安装说明，或者可供 plugin / CLI / MCP 使用的能力入口，系统就能把它纳入开发链路。

第一性原理 以实际 openclaw 为准 面向非专业开发者

OpenClaw 真正解决的是什么问题

问题不是“怎么读取一段说明文档”，而是“怎么让 agent 在真实运行时里稳定获得一种新能力”。这意味着系统必须处理 发现、优先级、依赖、配置、session 记忆、prompt 预算、安全边界和工具调用。

为什么 Agent-Reach 是一个好样本

因为它不是只有 README。它同时带有 agent_reach/skill/SKILL.md、references 和真正的外部工具安装路径， 所以它既有“知识层”，也有“能力层”。

为什么必须讲运行时而不是只讲 skill

单独讲 skill 会让人误以为“写一个 SKILL.md 就够了”。源码显示不是这样。skill 要想真正工作，背后还要接到 tool、plugin、gateway、workspace 和 session 管理上。

这份讲解会刻意避免把 OpenClaw 神秘化。我们不用类比，不把它说成“像某某平台”。 直接还原事实：仓库里有什么文件，OpenClaw 在哪里扫描，怎么过滤，哪些内容被写入 session，模型实际拿到了什么。

Repo Anatomy

GitHub 仓库到底向 OpenClaw 提供了什么

以 learning proj/Agent-Reach 为例，仓库的价值不是“在 GitHub 上”，而是里面放着 OpenClaw 可以吃进去的结构化资产。只要这些资产存在，仓库就不只是静态文档，而是能进入 agent 运行时的数据源。

第 1 类资产：skill 知识层

在 agent_reach/skill/SKILL.md 里，仓库声明了 skill 的名称、用途、触发场景和 references。 这决定了 agent 在什么任务里会考虑调用它，以及调用时该读哪些说明。

这一步很关键，因为它解决的是“模型知道什么”而不是“模型能执行什么”。如果没有这一层，外部工具即使已经安装，模型也不一定知道何时该用、先做哪一步、失败后如何回退。

可查文件
  • learning proj/Agent-Reach/agent_reach/skill/SKILL.md
  • learning proj/Agent-Reach/agent_reach/skill/references/*.md

第 2 类资产：安装与能力入口

Agent-Reach 的 README 明确写了安装链路，不只是复制一个 skill 目录，还包括安装 CLI、准备系统依赖、配置搜索能力和注册 SKILL.md。

这说明 skill 扩展不是“把一段 Markdown 丢进去”这么简单。真正能让它落地的是：仓库同时给出了能力入口和安装说明，所以 agent 或操作者可以把说明转成可执行环境。

可查文件
  • learning proj/Agent-Reach/README.md：安装步骤里明确提到注册 SKILL.md。
  • learning proj/Agent-Reach/README.md：列出 gh、yt-dlp、mcporter 等外部能力。
所以，当别人说“给一个 GitHub 链接就能安装 skill”时，准确展开后应该是： 这个链接指向了一个包含 skill 资产、安装文档、外部工具依赖和可能的 plugin / MCP 入口的仓库； OpenClaw 负责把这些资产接入自己的扩展机制，而不是凭空理解 GitHub 平台本身。

Runtime Design

运行时如何把这些模块拼起来

这一节把“模块”连成“运行系统”。重点不是单个函数，而是多次运行之间怎么保持一致性，怎么控制 prompt 体积，怎么把会话上下文和扩展能力一起交给模型。

1. 会话开始时，先做能力快照

在 agent-command.ts 里，新 session 或旧 session 没有 skillsSnapshot 时，会构建 workspace skill snapshot，并把结果写回 session entry。这样后续运行不必每次重新扫描全量 skill。

2. 快照不是缓存副本，而是运行边界

快照里保留的是当前 session 已经认领过的 skill 视图。它的意义不只是加速，更重要的是让同一会话内的能力集合保持一致，不因为磁盘上的 skill 目录变化而随时漂移。

3. prompt 注入有预算，不是全文硬塞

src/agents/skills/workspace.ts 里会处理 prompt 预算、compact mode 和截断提示。也就是说，OpenClaw 明确知道“skill 太多会挤占模型思考空间”，所以需要做裁剪。

4. 安装配置和运行配置分开

安装阶段解决“依赖有没有”；运行阶段解决“本次 run 看到了什么、允许用什么工具、在哪个 channel 回消息、沿用哪个 session”。 这就是为什么 skill 配置、plugin 配置、session 配置分散在不同地方。

5. tool 与 channel 把能力送到现实世界

skill 负责告诉模型怎么做，tool 负责真的执行，channel 负责和外部会话空间连接。比如一个子 agent 被 thread 绑定到某个 channel，它不只是“跑了一个模型”，而是在一个可追踪的会话上下文里运行。

6. 为 subagent / cron 留出的空间

从源码能看到 OpenClaw 已经非常重视 session key、channel 绑定、workspace 继承和 spawn 语义。这意味着它的运行时不是一次性脚本，而是朝着多轮、多会话、多执行主体的系统设计靠近。这一点对你后续做多 agent 编排很重要。

把“loop”讲透：为什么 agent 不是一次请求就结束
一次 agent run 的过程，至少包括：接收当前消息、读取 session 现状、解析本次可用 skill 视图、生成系统提示、 让模型决定是否调用工具、接收工具结果、继续下一轮推理，最后才输出文本或发送到 channel。 所以“循环”不是抽象词，而是运行时一轮轮把状态推进的机制。
把“记忆”讲透：这里的记忆至少有三层
第一层是 session entry，本轮会话的配置和 snapshot 会放这里。第二层是 transcript，记录这次会话之前发生过什么。 第三层是 memory plugin 层，文档里能看到 memory-core、memory-lancedb 这样的 plugin slot。也就是说，OpenClaw 不是只有一种“记忆”，而是短期会话状态和长期记忆插件并存。

  • learning proj/openclaw/src/agents/pi-embedded-runner/skills-runtime.ts：运行时优先使用 snapshot。
  • learning proj/openclaw/docs/tools/plugin.md：plugin slot 里有 memory 类插件。
Boundary

skill、tool、plugin、channel 的边界

这一节很重要，因为很多误解都来自把这四个词混成一个东西。它们互相合作，但职责并不一样。

skill：操作知识层

skill 主要由 SKILL.md 和 references 组成。它告诉 agent：遇到什么问题时考虑我、我依赖什么、先做什么后做什么、必要时去读哪些额外说明。

tool：实际执行层

tool 负责读文件、跑命令、发请求、写结果、调用外部系统。没有 tool，skill 只有知识没有动作。源码里还能看到对危险工具和自动批准范围的控制。

plugin：能力供应层

plugin 可以提供新的 channel、模型 provider、memory、speech、tools、skills 等能力。它是“系统能长出什么新器官”的地方，不只是一个打包格式。

channel：会话投递层

channel 决定消息从哪里来、回到哪里去、thread 如何绑定。对多 agent 或外部平台集成来说，这一层决定了会话是怎么落地的。

一个最容易说错的话是：“skill 就是插件。”严格说不对。 skill 更像被加载进 runtime 的操作知识；plugin 是系统扩展包，能新增 skill，也能新增 tool、memory、channel、provider。

  • learning proj/openclaw/src/agents/skills/plugin-skills.ts：plugin 提供的 skill 目录会被收进 skill 扫描流程。
  • learning proj/openclaw/src/acp/client.test.ts：可看到 read/search 类工具与危险工具的审批边界。
Decision Guide

给 AI vibe coder 的开发决策建议

如果你后面要设计自己的 skill 仓库，或者想把 agent-alpha 做成能稳定安装类似 Agent-Reach 这类 skill 的系统，下面这些判断标准比“能不能跑通一次”更重要。

先决定仓库输出什么资产

  • 如果只是行为指导，就产出稳定的 SKILL.md 和 references。
  • 如果还要真正执行任务，就同时给出 CLI / MCP / 插件入口和安装说明。
  • 如果需要长期维护，最好让 skill 结构与代码结构分离，不要把所有逻辑挤进一个 Markdown。

把“是否能被运行时接住”放在第一位

  • 能否放进 OpenClaw 已扫描的 skill 根目录。
  • frontmatter 是否足够声明依赖、启用条件和安装方式。
  • 依赖安装后，工具路径是否真的可被 host runtime 访问。

控制 prompt 体积

  • 不要把 skill 写成无边界长文。
  • 主说明放在 SKILL.md，细节放到 references。
  • 因为 OpenClaw 最后会做预算裁剪，所以结构清晰比堆字更重要。

为未来的多 agent 编排做准备

  • 不要把 skill 绑定死在某个唯一工作目录。
  • 把环境变量、api key、外部工具路径做成可配置项。
  • 让 session、workspace、channel 的关系可解释，这样日后才好接 subagent 和 cron。
Talk Track

演示时可以怎么讲

这一节不是新知识，而是把上面的技术内容压缩成适合口头表达的版本。适合你向老板或团队顺着讲，不容易跑偏。

一句话版

OpenClaw 不是“会安装 GitHub”，而是有一套能把仓库里的 skill 资产、安装说明和工具入口纳入 agent 运行时的机制。

三句话版

  1. GitHub 链接只是入口，真正生效的是仓库里的 skill 资产和依赖安装链路。
  2. OpenClaw 会扫描、过滤、快照并注入 skill，而不是把整仓库直接塞给模型。
  3. skill 只负责行为知识，真正执行依赖 tool、plugin、channel 和 session 运行时。

如果老板问“为什么要设计这么多层”

因为只解决“读到说明”不够，系统还必须解决：权限、依赖、上下文复用、会话一致性、工具调用、安全边界和跨轮执行。 少一层，这套扩展机制都会变脆。

如果别人问“那 GitHub 链接本身值不值钱”

值钱，但值钱的原因不是它是 GitHub，而是它把仓库内容暴露成了 agent 可以读取和安装的入口。没有结构化资产，链接本身没有魔法。

整体结论： 对 OpenClaw 来说，skill 扩展不是一个单点功能，而是一整套“发现能力、判断能力、安装依赖、快照能力、把能力注入运行时、在会话里持续复用”的系统机制。 所以你未来设计新扩展仓库时，重点不该是“怎么把链接发给 agent”，而该是“仓库里的资产能否被这套运行时稳定接住”。
