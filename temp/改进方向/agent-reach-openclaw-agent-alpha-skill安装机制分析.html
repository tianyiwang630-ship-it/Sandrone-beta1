Harness Engineering Report

为什么 OpenClaw 能装上 Agent Reach，而 agent-alpha 还不行

这不是“模型会不会自己悟”的差别，而是 harness 工程能力链条的差别。
Agent Reach 本质上不是一个普通 prompt skill，而是一套“安装脚手架 + CLI + 外部依赖 + 最终 skill 注册”的复合型能力包。 OpenClaw 已经具备承接这类能力包的外壳，而 agent-alpha 目前还把 skill 主要当成文档资源。

一句话结论 OpenClaw 能装，是因为它有“安装闭环”；agent-alpha 只有“读取闭环”。
关键对象 Agent Reach 不是普通 skill，而是带安装器的 skill scaffold。
借鉴方向 把 skill 从“说明书”升级成“可安装能力包”。

1. 先说结论

如果从 harness engineering 角度拆开看，这个问题可以归结为三个层次：对象是不是可安装单元、宿主有没有安装机制、安装后有没有运行与隔离机制。

OpenClaw

已经把 skill 当成一等公民。skill 可以有依赖声明、安装策略、OS 限制、工具目录、安装状态和安全边界。

Agent Reach

本身就自带 CLI 安装器，会装系统依赖、会配置外部工具、最后还会把自己的 SKILL.md 安装到 OpenClaw 的 skill 目录。

agent-alpha

目前把 skill 主要当成“可被 prompt 加载的说明文档”。它能发现 skill，但还不会把 skill 当成“需要安装、校验、托管”的能力包。

2. Agent Reach 到底是什么

它不是普通 skill

  • 它有自己的 Python 包和 CLI 入口，`pyproject.toml` 里注册了 `agent-reach = agent_reach.cli:main`。
  • 它不是只放一个 `SKILL.md`，而是把 `guides/`、`skill/`、`scripts/` 一起打进安装包。
  • 它的 `install` 命令会先安装依赖，再配置环境，最后才把 skill 注册给宿主 agent。
  • 它的 `SKILL.md` 也依赖外部 CLI，例如 `mcporter`、`gh`、`curl`、`twitter`、`yt-dlp` 等。
换成更直白的话 Agent Reach 不是“给 agent 看的文档”，而是“给 agent 装能力的脚手架”。
最终让模型调用的是 skill 文档，但前面真正起作用的是安装器、CLI、依赖和环境配置。
Agent Reach CLI 里直接提供： - install - doctor - configure - uninstall - skill --install / --uninstall
_cmd_install() 的流程是： 1. 创建 ~/.agent-reach/tools 2. 安装核心系统依赖 3. 安装 / 配置 mcporter 4. 安装可选渠道依赖 5. 导入 cookies 6. doctor 检查 7. 调用 _install_skill()

3. OpenClaw 为什么能装上 Agent Reach

OpenClaw 能装上，并不是因为“它更聪明”，而是因为它的 harness 已经把下面这条链条打通了。

  1. 它承认 skill 不是纯文档。 OpenClaw 的 skill metadata 有 `requires` 和 `install` 概念，说明 skill 可以声明自己缺什么、怎么装。
  2. 它承认安装器是 skill 生命周期的一部分。 `install` 支持 `brew`、`node`、`go`、`uv`、`download` 多种类型。
  3. 它有执行安装动作的能力。 OpenClaw 本身可以运行安装命令，不是只会加载 prompt 文本。
  4. 它有每个 skill 自己的工具目录。 下载或解压的内容不会乱写到项目根目录，而是进入独立 tools root。
  5. 它有安全边界。 下载和解压会检查路径越界，安装前还会做代码安全扫描。
  6. 它有托管 skill 的目录体系。 `~/.openclaw/skills/...` 这类目录是 OpenClaw 的正式 skill 挂载点。
  7. 它还能配合 Agent Reach 自己的安装器。 当你把 Agent Reach 的安装文档或仓库线索给到 OpenClaw，它既能执行里面的安装步骤，也能接住最后落下来的 skill 文件。
能力环节 OpenClaw 的现状 对 Agent Reach 成功安装的意义
Skill 元数据模型 有 `requires`、`install`、`os`、`skillKey`、`homepage` 等字段 能把“这个 skill 需要外部命令和安装器”表达出来
安装器类型 支持 `brew`、`node`、`go`、`uv`、`download` 能够覆盖常见 CLI、二进制工具、工具包下载等场景
状态判断 能判断缺哪些 bin、env、config，并挑选安装方案 不是盲装，而是“先判断缺口，再执行安装”
工具隔离 每个 skill 有独立的 tools 目录 下载的 exe、解压包、辅助工具有正式归属地
安装安全 下载有 SSRF 防护，解压检查路径边界，安装前做扫描 减少“安装一个 skill 却把整个环境搞脏”的风险
宿主目录接入 OpenClaw 识别 `~/.openclaw/skills` 等技能目录 Agent Reach 安装完可以把自己的 skill 正式挂进去

4. Agent Reach 是怎么把自己“装进 OpenClaw”的

Agent Reach 自己做的事

  • 运行 `agent-reach install` 时，会先准备 `~/.agent-reach/tools`。
  • 会安装系统依赖和渠道依赖，而不是只复制一份说明文档。
  • 最后调用 `_install_skill()`，把 `SKILL.md + references/` 复制到宿主 skill 目录。
  • 它明确检查 `~/.openclaw/skills`，也支持 `OPENCLAW_HOME` 指向的 OpenClaw 目录。

OpenClaw 宿主做的事

  • 允许 agent 执行安装说明里的命令，前提是工具权限配置到位。
  • 有正式的 skill 发现与加载机制，能在 `~/.openclaw/skills/agent-reach` 看到它。
  • 有安装状态与依赖模型，未来还能继续托管和诊断这种 skill。
这里最容易误判的一点 不是 OpenClaw “凭空把一个仓库变成 skill”。
真正发生的是：Agent Reach 自己本来就是一个带安装逻辑的包；OpenClaw 则是一个足够成熟的宿主，能执行安装、承接结果、识别并继续使用这个 skill。

5. 为什么 agent-alpha 现在做不到

agent-alpha 现在不是“差一点”，而是能力链条还停留在更前面的阶段。它已经会发现 skill，但还没有进入 skill 生命周期管理。

维度 OpenClaw agent-alpha 结果
skill 定义 能力包，可带依赖与安装策略 主要是 `SKILL.md` 文档 + 可选脚本 agent-alpha 不知道“这个 skill 需要先装”
前置检查 判断 bin / env / config 是否缺失 没有正式的依赖检查层 模型即使看到了 skill，也不知道缺口在哪
安装器模型 内建 `brew/node/go/uv/download` 没有 skill install spec 不能把“安装动作”结构化
工具目录 每个 skill 有独立 tools root 没有 per-skill artifact 目录 exe、zip、CLI 工具无正式存放位置
宿主目录体系 有 shared/workspace/managed 技能目录 主要盯着项目根目录 `skills/` 不适合装“外来 skill”或多实例 skill
安全安装边界 已有下载、扫描、路径边界处理 没有面向 skill 安装的安全层 就算硬做安装，也不够稳
agent-alpha/agent/core/skill_loader.py 的核心逻辑： 1. 递归找 SKILL.md 2. 解析简单 frontmatter 3. 给系统提示词做 skill summary 4. 需要时返回 skill body 也就是说，它是在“读 skill”，不是“装 skill”。
agent-alpha/agent/core/tool_loader.py 的核心逻辑： 1. 加载 MCP tools 2. 加载 shared load_skill 3. 加载 builtin tools 这里也没有“managed skill installer”这一层。

6. OpenClaw 最值得借的，不是界面，而是这套机制

机制 1

Skill 元数据升级

不再只有 `name/description/tags`，而要允许 skill 声明 `requires`、`install`、`os`、`homepage`、`artifactDir` 等信息。

机制 2

安装状态层

在真正调用 skill 之前，先回答三个问题：缺什么、能不能装、建议怎么装。这样模型就不是瞎试。

机制 3

每个 skill 自己的工具目录

任何下载的 exe、zip 解压内容、CLI 工具缓存，都应该有独立归宿，而不是散落在项目根目录或工作区里。

机制 4

受控安装器执行

不是让模型随便拼 bash，而是让 harness 认识有限几类安装方式，并统一处理错误、回退和权限。

机制 5

外来 skill 接入路径

要能从本地目录、仓库、包管理器、下载链接等来源安装 skill，而不是只能读项目自带的 `skills/`。

机制 6

安全边界

下载地址、解压目录、可执行文件位置、安装后扫描，这些都要由 harness 接手，不能完全丢给模型的自由发挥。

7. agent-alpha 如何支持这种“非典型 skill”

这里的“非典型 skill”，指的不是只有 `SKILL.md` 的 skill，而是可能携带 exe、CLI、MCP server、下载包、系统依赖、配置向导的一类能力包。

1

先把 skill 定义改对

  • 给 `SKILL.md` frontmatter 增加结构化字段，或者单独引入 `skill.json`。
  • 最少应支持：`requires.bins`、`requires.env`、`install[]`、`os`、`skill_key`、`entrypoints`。
  • 把“普通文档 skill”和“安装型 skill”统一纳入一个模型。
2

再补一个 skill manager

  • 负责发现 skill、计算状态、执行安装、登记产物、暴露错误信息。
  • 不要把这层塞进 `skill_loader.py`，因为那会把“读文档”和“装能力”搅在一起。
  • 这层应该比 `load_skill` 更靠外，属于 harness 控制面。
3

最后接入运行时

  • 当模型要调用某 skill 时，运行时先看它是否 ready。
  • 如果没装好，优先走受控安装流程，而不是让模型自己随手写命令。
  • 安装完成后，再把 skill 能力暴露给提示词和工具系统。

8. 一个适合 agent-alpha 的落地模型

建议的数据模型

SkillPackage - id - name - source (local/git/url/package) - workspace_scope / agent_scope - docs (SKILL.md, references/) - requires - bins[] - env[] - config[] - install[] - kind: uv | node | go | download | pip | exe | mcp - payload - artifacts_dir - status - discovered - installable - installed - ready - failed

建议的控制流

discover skill -> parse metadata -> evaluate requirements -> choose install strategy -> install into per-skill dir -> register binaries / MCP / docs -> expose skill summary to prompt -> allow model invocation
这里最关键的一步 不要再让 “skill 是否可用” 只取决于 `SKILL.md` 是否存在。
真正合理的条件应该是：文档存在、依赖满足、安装完成、产物可达、运行时已登记。

9. 分阶段改造建议

第一阶段：最小闭环
  • 给 skill frontmatter 增加 `requires` 和 `install`。
  • 新增 `skill_manager.py`，只做发现、状态计算、安装入口。
  • 给每个 skill 分配独立 artifacts 目录，比如 `/tools//`。
  • 先支持 `uv`、`node`、`download` 三种安装类型就够了。
第二阶段：适配 Agent Reach 这类 skill
  • 支持“skill 安装后还要注册 SKILL.md”的模式。
  • 支持 repo/url/package 作为 skill 来源。
  • 允许安装器声明“安装完成后执行注册动作”，例如复制文档、登记 MCP、写配置。
  • 把 `doctor` 之类的检查结果也纳入 skill status。
第三阶段：真正进入 orchestrator 能力
  • 把 skill 安装与 agent 实例目录绑定，而不是绑项目根目录。
  • 支持多 agent / 多 workspace 的 skill 隔离与共享策略。
  • 给前端或 API 暴露 skill 状态、安装日志、失败原因。
  • 加上安全审计和受控权限模型，避免变成“万能执行器”。

10. 最后的判断

判断 1

OpenClaw 能装 Agent Reach，不是偶然成功，而是因为 OpenClaw 和 Agent Reach 正好在同一套“可安装 skill”范式里。

判断 2

agent-alpha 目前还停留在“skill 文档分发”范式，所以面对这种带 CLI / exe / 安装器的 skill，会天然吃力。

判断 3

你真正该借鉴的不是 OpenClaw 的所有复杂度，而是它把 skill 从“说明书”升级成“生命周期对象”的那层设计。
