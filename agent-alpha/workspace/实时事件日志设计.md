# 实时事件日志设计

日期：2026-06-06

## 1. 背景

当前 `agent-alpha` 的两类会话记录分别是：

- `session-log/sessions/*.json`：会话快照，主要用于恢复会话。
- `session-log/logs/*.json`：会话归档，主要在退出、手动保存或中断退出时生成。

这两类记录都不是“运行中实时可看”的日志。

当前问题是：用户发出消息后，如果 agent 进入工具调用过程，必须等到本轮结束、双击 `Esc` 中断，或者退出时，才能在文件里看到更新。这样不利于观察 agent 当前卡在哪个工具、工具返回了什么、执行链路进行到了哪一步。

## 2. 目标

新增一套实时事件日志，让用户在 agent 运行过程中就能看到新增记录，而不是等一轮结束。

第一版目标很明确：

- 不改 `sessions/*.json` 的职责。
- 不改 `logs/*.json` 的职责。
- 只新增 `events/*.jsonl`，负责实时追加。
- 让用户在工具调用过程中，直接从文件看到 `history` 的新增项。

## 3. 不做什么

第一版不做这些事：

- 不重构现有 `sessions` 和 `logs` 持久化逻辑。
- 不让 `events` 反向生成 `sessions` 或 `logs`。
- 不额外设计一套复杂事件模型。
- 不为 subagent、cron 单独设计新格式。
- 不做数据库、索引、检索或 UI 展示层。

## 4. 核心设计

### 4.1 新增事件目录

新增目录：

```text
agent-alpha/session-log/events/
```

每个会话一个文件：

```text
agent-alpha/session-log/events/<session_id>.jsonl
```

文件扩展名继续使用 `.jsonl`，但内容不是严格的“单行一条”。

实际写入格式为：

- 每次追加一个格式化后的多行 JSON 块；
- 事件与事件之间空一行；
- 仍然保持 append-only，不重写整文件。

这样做的原因：

- 继续保留实时追加的优点；
- 对人阅读明显更友好；
- 比完整维护一个大数组 JSON 更简单。

### 4.2 记录内容直接沿用 history

第一版不额外发明新的“事件语义层”。

实时日志直接记录当前会话 `history` 中新增的项，也就是沿用现有结构：

- `user`
- `assistant`
- `tool`

只要代码里出现新的 `history.append(...)`，就同步把这一项追加到 `events/<session_id>.jsonl`。

这样做的理由：

- 最简单；
- 与当前运行时结构一致；
- 用户看到的内容和最终会话历史一致；
- 不需要维护两套平行数据模型。

### 4.3 每条记录结构

每次追加的 JSON 块不是裸写 history 项，而是加一个很薄的外壳：

```json
{
  "ts": "2026-06-06T18:00:00",
  "seq": 12,
  "session_id": "6bcc05",
  "entry": {
    "role": "assistant",
    "content": "",
    "tool_calls": []
  }
}
```

字段说明：

- `ts`：写入时间。
- `seq`：当前会话内递增序号，用来稳定表达先后顺序。
- `session_id`：当前会话 id。
- `entry`：原始 `history` 项，不改 role 结构。

示意上，一个文件内会是这样的排版：

```json
{
  "ts": "2026-06-06T18:00:00",
  "seq": 12,
  "session_id": "6bcc05",
  "entry": {
    "role": "assistant",
    "content": ""
  }
}

{
  "ts": "2026-06-06T18:00:02",
  "seq": 13,
  "session_id": "6bcc05",
  "entry": {
    "role": "tool",
    "tool_call_id": "call_xxx",
    "content": "..."
  }
}
```

### 4.4 能看到什么

采用这个设计后，用户在运行中可以看到：

- 自己刚发出的 `user` 消息；
- assistant 发起工具调用前写入的 `assistant` 项；
- 工具返回时写入的 `tool` 项；
- 最终回答写入的 `assistant` 项；
- 中断场景下补入的工具占位项。

换句话说，只要某条内容已经进入 `history`，就应该同步出现在 `events/*.jsonl`。

## 5. 工具结果截断规则

第一版保留“尽量原文记录工具结果”的原则，但需要加上上限，避免单个工具输出把实时日志写得过大。

约定如下：

- 单个工具结果最多记录 `200` 行；
- 同时增加字符上限；
- 超过任一上限就截断；
- 截断后仍然写入 `events`，但要明确标记该条结果已经截断。

建议在 `entry` 外层增加截断元信息，例如：

```json
{
  "ts": "2026-06-06T18:00:00",
  "seq": 15,
  "session_id": "6bcc05",
  "truncated": true,
  "original_line_count": 356,
  "original_char_count": 78214,
  "entry": {
    "role": "tool",
    "tool_call_id": "call_xxx",
    "content": "..."
  }
}
```

说明：

- `entry` 里仍然保留当前实际写入的截断后内容；
- 原始总行数、总字符数放在外层，便于判断是否被截断；
- 对非 `tool` 项，默认不需要加这组字段。

## 6. 与现有记录的分工

三套文件的职责保持清晰分离：

- `sessions/*.json`：会话快照，用于恢复。
- `logs/*.json`：会话归档，用于完整留档。
- `events/*.jsonl`：运行中实时追加，用于边跑边看。

`events` 不是新的主记录源，只承担“可观察性”职责。

## 7. 推荐落盘位置

第一版推荐在真正追加 `history` 的地方同步写 `events`，而不是在 CLI 外围额外猜测状态。

这样可以保证：

- 写入时机和 `history` 一致；
- 不容易漏；
- 中断补写逻辑也能自然覆盖；
- 未来 subagent、cron 复用同一套 runtime / loop 时，更容易沿用。

实现层面上，优先考虑：

- 在会话开始时初始化当前 `session_id` 对应的 event writer；
- 在所有 `history.append(...)` 的实际落点旁边同步追加 `events`。

## 8. 第一版成功标准

完成后，应满足这些可验证结果：

1. 用户发出消息后，不等最终回答，就能在 `events/<session_id>.jsonl` 里看到新的 `user` 项。
2. assistant 发起工具调用时，能先看到对应的 `assistant` 项。
3. 工具执行结束后，能立刻看到对应的 `tool` 项。
4. 如果工具结果过大，`events` 中写入截断后的内容，并明确标记已截断。
5. `sessions` 和 `logs` 的现有行为保持不变。

## 9. 当前结论

这版设计选择的是最小改动路线：

- 不重做日志系统；
- 不引入复杂事件模型；
- 直接把 `history` 的新增项实时旁路写入 `events/*.jsonl`；
- 只额外补上必要的时间、顺序和截断信息。

这足以满足当前目标：用户在 agent 运行过程中，可以直接从文件观察它正在做什么。
